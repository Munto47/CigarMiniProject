## 1. 测试金字塔与覆盖率要求

```
        ┌──────────┐
        │   E2E    │  10%   关键链路（充值、下单、退款）
        │（沙箱）  │
        ├──────────┤
        │ 集成测试 │  30%   Service+真 PG/Redis（含事务、幂等）
        ├──────────┤
        │ 单元测试 │  60%   Service 纯逻辑、工具函数
        └──────────┘
```

| 模块 | 单元覆盖率门槛 | 集成覆盖率门槛 |
|---|---|---|
| `payment / refund / recharge / member` | **≥ 90%** | **≥ 80%** |
| `order / cart / cigar` | ≥ 85% | ≥ 75% |
| `recommend / review / library` | ≥ 75% | ≥ 60% |
| 其他 | ≥ 70% | — |

**强制门禁**：CI 跑覆盖率，低于门槛则不能合并。

## 4. 并发与压力测试

> 工具：k6 / Jest 多 promise + testcontainers。

### 4.1 关键并发场景

#### TC-CONCUR-001：库存超卖防御

```ts
// 准备：cigar.stock=100, locked=0
// 200 个并发请求，每个 qty=1
const promises = Array.from({ length: 200 }, () =>
  request(app).post('/api/orders').set('Idempotency-Key', uuid()).send({ items:[{cigarId, qty:1}] })
);
const results = await Promise.allSettled(promises);
const success = results.filter(r => r.status === 'fulfilled' && r.value.body.code === 0).length;

expect(success).toBe(100);                                // 100 个成功
expect((await prisma.cigar.findUnique({where:{id:cigarId}})).stockLocked).toBe(100n);
```

预期：100 成功，100 失败（3002 库存不足），无超卖。

#### TC-CONCUR-002：余额双扣防御

```ts
// 准备：balance=10000，订单 amount=10000
// 同一订单同时 2 个 pay 请求（不同 idemKey 模拟前端 bug）
const [r1, r2] = await Promise.allSettled([
  request(app).post(`/api/orders/${id}/pay`).set('Idempotency-Key', 'k1').send({}),
  request(app).post(`/api/orders/${id}/pay`).set('Idempotency-Key', 'k2').send({}),
]);
```

预期：1 成功 + 1 失败（4002 状态非 pending；NOWAIT 抢锁失败也可，前端重试时会进幂等）。最终余额 = 0。

#### TC-CONCUR-003：充值回调重放

```ts
// 同 transaction_id 微信回调 50 次
const promises = Array.from({ length: 50 }, () => callWechatCallback(payload));
await Promise.allSettled(promises);

const callbacks = await prisma.paymentCallback.findUnique({ where: { /* unique */ } });
expect(callbacks.receivedCount).toBe(50);
const profile = await prisma.memberProfile.findUnique({ where: { userId } });
expect(profile.balanceCents).toBe(10000n);                // 仅入账 1 次
```

#### TC-CONCUR-004：管理员手动调余额双击

```ts
const idemKey = uuid();
const [r1, r2] = await Promise.allSettled([
  adjustBalance({ userId, amount: 100, idemKey }),
  adjustBalance({ userId, amount: 100, idemKey }),
]);
```

预期：1 成功 1 幂等命中（返回相同 transactionId），余额仅 +100。

#### TC-CONCUR-005：退款超额防御

```ts
// 订单 actual_pay=200, refunded=0
// 2 个管理员同时各发起 150 元退款
const [r1, r2] = await Promise.allSettled([
  refund({ orderId, amount: 150, idemKey: 'k1' }),
  refund({ orderId, amount: 150, idemKey: 'k2' }),

```

预期：1 成功，1 抛 4003 或 CHECK 约束错。最终 refunded ≤ 200。

## 5. 支付沙箱端到端

### 5.1 微信支付沙箱

> 沙箱要专用 mchid。线下走通后再切生产。

```
Step 1: 调 POST /api/member/recharge { tierId: 100元 }
        → 返回 payParams
Step 2: 用微信开发者工具调 wx.requestPayment(payParams)
        → 选「沙箱模拟支付成功」
Step 3: 等微信沙箱回调（一般 5~30 秒）
        → 服务端 payment_callbacks 应有一条记录
        → recharge_orders.status='success'
        → member_profiles.balance_cents 增加
Step 4: 用 API 查询 GET /api/member/transactions
        → 应有一条 type=recharge, amount=10000
```

### 5.2 微信退款沙箱

```
Step 1: 用沙箱跑通一笔订单（先小程序下单微信支付）
Step 2: 管理后台发起退款 POST /api/admin/orders/{id}/refund
Step 3: 等微信退款回调
Step 4: 验证 refund_records, balance_transactions, orders.refunded_amount_cents
```

### 5.3 美团沙箱

```
Step 1: 管理后台开单（pay_method=meituan）
Step 2: mock 美团下单 API（接美团测试环境）
Step 3: 美团 mock 回调
Step 4: 验证 orders.status='paid', stock 实扣, 消费积分增加
```

### 5.4 故障模拟

| 故障 | 模拟方法 | 预期行为 |
|---|---|---|
| 微信回调丢失 | 沙箱开关，禁用回调 | 5 分钟内 cron 兜底入账 |
| 微信下单超时 | 防火墙 drop 请求 | 6001，前端重试或换支付方式 |
| 美团回调丢失 | 同上 | 10 分钟内 cron 兜底，否则 out_of_sync 告警 |
| DB 突然 down | 杀 PG 进程 | 业务返回 9001，30 秒内自愈 |
| Redis down | 杀 Redis | 幂等 SETNX 走不到 → DB 唯一约束兜底 |
