## 1. 三道幂等防线

数据完整性靠三道防线，缺一不可：

```
┌──────────────────────────────────────────────────────┐
│  防线 1：客户端 Idempotency-Key + Redis SETNX        │  → 防 99% 双击
│  防线 2：业务表 idempotency_key 字段 UNIQUE 约束     │  → 防 Redis 失效/重启
│  防线 3：流水表 (user_id, related_no, type) UNIQUE   │  → 防回调重放/服务异常
└──────────────────────────────────────────────────────┘
```

### 1.1 业务表的幂等键

```sql
-- orders 表
idempotency_key VARCHAR(64) NOT NULL,
CONSTRAINT uniq_orders_idem UNIQUE (user_id, idempotency_key)

-- recharge_orders 表（部分唯一索引，因为 cron 内部创建的可不带 key）
CREATE UNIQUE INDEX uniq_recharge_idem
  ON recharge_orders(user_id, idempotency_key)
  WHERE idempotency_key IS NOT NULL;

-- refund_records 表
CREATE UNIQUE INDEX uniq_refund_idem
  ON refund_records(order_id, idempotency_key)
  WHERE idempotency_key IS NOT NULL;
```

### 1.2 流水表的业务事件唯一

```sql
-- 同一笔充值/订单/退款，对同一用户、同一类型，只能产生一条流水
CREATE UNIQUE INDEX uniq_bt_event
  ON balance_transactions(user_id, related_type, related_no, type);

CREATE UNIQUE INDEX uniq_pt_event
  ON point_transactions(user_id, level_type, related_type, related_no, type);
```

**为什么 `related_no` 要参与唯一**：
- 回调表 `(channel, external_id)` 唯一已经能挡住一次回调的重放；
- 但流水表自身的唯一约束是兜底——避免「不同回调通道但同一业务单」的边界情况，比如「美团回调失败 → 走人工补单 → 用 `payment_callbacks` 模拟一次」时，流水表能再挡一次。

## 2. 为什么不存 `cigars.ai_tags`（数组）

评审决定：**方案 B**——保留 `cigar_tags + flavor_tags`，删除 `cigars.ai_tags`。

| 维度 | 方案 A（仅 ai_tags） | 方案 B（关联表 + 标签库）✅ |
|---|---|---|
| 模型表达力 | 弱：标签无权重 | 强：每个标签有 `ai_weight + score_map` |
| 标签管理 | 散落在每条 cigar 行 | 集中在 `flavor_tags` 表，管理后台可 CRUD |
| 推荐算法 | 只能做匹配命中 | 可基于权重 + 风味分数加权 |
| 一致性 | 拼写错误会蔓延 | 引用唯一标签 ID |

代价：查雪茄需 JOIN，但 `cigar_tags` 复合主键已经覆盖。

**推荐评分公式**（详见 `03_关键场景实现指南.md §6`）：

```
score(cigar) = Σ tag.ai_weight × match(answer, tag.score_map)
```

## 3. `level_configs` EXCLUDE 约束的细节

```sql
ALTER TABLE level_configs
  ADD CONSTRAINT no_overlap_levels EXCLUDE USING GIST (
    level_type WITH =,
    int8range(min_points, COALESCE(max_points, 9223372036854775807), '[]') WITH &&
  );
```

含义：对同一 `level_type`（充值或消费），不允许两条记录的积分区间 `[min_points, max_points]` 重叠。

### 3.1 为什么用 `int8range` + `'[]'` 闭区间

业务约定：V1 = `[0, 999]`，V2 = `[1000, 1999]`。`'[]'` 表示双闭区间，符合「999 属于 V1，1000 属于 V2」的直觉。

### 3.2 为什么用 `9223372036854775807` 而不是 NULL

`int8range(0, NULL)` 在 PG 中是无效区间。`9223372036854775807` 是 `BIGINT.MAX`，等价于「无上限」。

### 3.3 不能强制的约束（应用层处理）

DDL 只能保证「不重叠」，不能保证：
- V1 必须从 0 开始；
- 不能有断档（如 V1=0~999, V3=2000~2999, 缺 V2）；
- 区间必须连续。

所以 `LevelConfigService.update(...)` 必须在事务内额外校验：
```ts
const all = await tx.levelConfig.findMany({ where: { levelType }, orderBy: { level: 'asc' } });
if (all[0].minPoints !== 0n) throw new BusinessException(2001, 'V1 必须从 0 开始');
for (let i = 0; i < all.length - 1; i++) {
  if (all[i].maxPoints + 1n !== all[i + 1].minPoints) {
    throw new BusinessException(2001, `V${all[i].level} 与 V${all[i+1].level} 之间断档`);
  }
}
```

## 4. 库存预占字段的不变量

```
0 ≤ stock_locked ≤ stock        (CHECK 约束保证)
可售数量 = stock - stock_locked
```

### 4.1 创建订单时的扣减 SQL

```sql
UPDATE cigars
   SET stock_locked = stock_locked + $qty,
       updated_at = now()
 WHERE id = $cigarId
   AND stock - stock_locked >= $qty
RETURNING stock, stock_locked;
-- 影响行数 = 0 → 库存不足，业务异常 3002
```

注意：**WHERE 条件中包含 `stock - stock_locked >= $qty`**，这是关键。即便高并发，PG 行锁会让查询线性化，不会卖超。

### 4.2 支付成功时的「实扣」SQL

```sql
UPDATE cigars
   SET stock        = stock        - $qty,
       stock_locked = stock_locked - $qty,
       updated_at   = now()
 WHERE id = $cigarId;
-- 不变量保持：stock - stock_locked 在事务前后一致
```

### 4.3 取消/超时时的回退 SQL

```sql
UPDATE cigars
   SET stock_locked = stock_locked - $qty,
       updated_at   = now()
 WHERE id = $cigarId;
```

> **常见错误**：忘了在订单超时 cron 里回退 stock_locked。一个月后 `stock_locked` 累积，可售为 0，看起来"卖光了"实际卖不出。**测试用例 TC-007 必须覆盖**。

## 5. `payment_records` 不要 `refunded` 状态

### 5.1 为什么

部分唯一索引：
```sql
CREATE UNIQUE INDEX uniq_pay_order_success
  ON payment_records(order_id) WHERE status='success';
```

如果允许 `success → refunded`，那一笔订单退款后这条记录变成 `refunded` 不在唯一约束的 WHERE 范围内，**新的 success 可以再被插入**——重复支付。

### 5.2 正确做法

- `payment_records.status='success'` 是终态。
- 退款事实只记 `refund_records`。
- 是否被退款？查 `orders.refunded_amount_cents > 0`，或 `EXISTS (SELECT 1 FROM refund_records WHERE order_id=X AND status='success')`。

## 6. `orders.refunded_amount_cents` 字段的设计

### 6.1 为什么不用 SUM

```sql
-- 错误做法：
SELECT actual_pay_cents - COALESCE(SUM(amount_cents), 0)
  FROM orders o LEFT JOIN refund_records r ON r.order_id=o.id AND r.status='success'
 WHERE o.id=$1;
```

并发下：A 在算"还能退多少"时 B 已写一条 success，A 看不到 → 超额退款。

### 6.2 正确做法

事务内 += 字段，CHECK 约束兜底：

```sql
ALTER TABLE orders ADD COLUMN refunded_amount_cents BIGINT NOT NULL DEFAULT 0
  CHECK (refunded_amount_cents <= actual_pay_cents);

-- 退款事务内
SELECT actual_pay_cents, refunded_amount_cents FROM orders WHERE id=$1 FOR UPDATE;
-- 校验 refunded_amount_cents + $amount <= actual_pay_cents
UPDATE orders SET refunded_amount_cents = refunded_amount_cents + $amount WHERE id=$1;
-- 如违反 CHECK 约束 → DB 抛异常 → 事务回滚
```

CHECK 约束是兜底：即使代码逻辑写错，DB 也不让 refunded 超过 actual_pay。

## 7. `payment_records` 的 `status='closed'` 是什么

```
pending → closed
```

场景：用户创建订单后没支付，30 分钟超时被 cron 关闭。`orders.status='cancelled'`，对应这笔订单的 `payment_records`（如果已生成）置为 `closed`。

为什么不直接删？保留便于审计（用户曾尝试支付但未完成）。

## 8. 触发器维护 `cigars.rating_avg/rating_count`

```sql
CREATE TRIGGER reviews_rating_sync
AFTER INSERT OR UPDATE OF status, deleted_at OR DELETE ON reviews
FOR EACH ROW EXECUTE FUNCTION trg_update_cigar_rating();
```

### 8.1 为什么不在应用层算

应用层维护 `rating_avg` 容易漏：
- 评论被屏蔽时忘了重算；
- 软删时忘了重算；
- 多个 service 各自更新导致不一致。

数据库触发器是最后一道保险。

### 8.2 性能影响

- 每条评论 INSERT 触发一次重算；
- 重算是 `SELECT AVG WHERE cigar_id=X AND status='visible'`，有 `idx_reviews_cigar_status` 索引，毫秒级。
- 高并发评论场景（不太会发生）可优化为异步。

## 9. JSONB 字段的存储与查询

### 9.1 何时用 JSONB

- 结构化但 schema 不稳定（如 `flavor_scores: {果香:80, 木香:75, ...}`）；
- 字段数量动态（如 `payment_callbacks.raw_payload`）；
- 不会用作 JOIN 键。

### 9.2 不要把 JSONB 当万能袋

下面这些场景**不该**用 JSONB：

| 场景 | 应该用 |
|---|---|
| 商品的标签列表 | `cigar_tags` 关联表 |
| 订单的支付方式 | `pay_method` VARCHAR + CHECK |
| 用户的等级 | `recharge_level/consumption_level SMALLINT` |

JSONB 的代价：无法走 B-tree 索引，无法外键，无法 CHECK。

### 9.3 GIN 索引

如需按 JSONB 内字段查询，加 GIN 索引：

```sql
CREATE INDEX idx_xxx_jsonb ON xxx USING GIN (jsonb_field);
-- 然后 WHERE jsonb_field @> '{"key":"value"}' 走索引
```

## 10. 软删 vs 硬删

| 表 | 删除策略 |
|---|---|
| `users` | 硬删（GDPR/合规可能要求物理删除，但当前设计不开放删除接口；如要支持，加 `deleted_at`） |
| `admins` | 软删（保留 ID 用于 operation_logs 引用） |
| `cigars / drinks` | 软删（订单 snapshot 还引用 ID） |

| `cart_items` | 硬删（无审计需要） |
| `cigar_tags / pairings` | 硬删（关联表） |
| `orders / payment_records / refund_records / *_transactions / level_change_logs` | **永不删**（流水永久保留） |

## 11. 索引策略

### 11.1 必备索引

每张表至少要有：
- 主键索引（PG 自动建）；
- 外键字段索引（PG 不自动建，必须手动加）；
- 高频查询字段索引（如 `user_id, created_at DESC`）；
- 状态字段的部分索引（如 `WHERE status='pending'`）。

### 11.2 不要过度索引

每个索引都有写入代价。监控 `pg_stat_user_indexes` 找出零使用率的索引并删除：

```sql
SELECT schemaname, tablename, indexname, idx_scan, idx_tup_read
  FROM pg_stat_user_indexes
 WHERE idx_scan = 0
 ORDER BY pg_relation_size(indexrelid) DESC;
```

### 11.3 复合索引顺序

复合索引 `(a, b, c)` 能满足查询：
- `WHERE a=?`
- `WHERE a=? AND b=?`
- `WHERE a=? AND b=? AND c=?`

不能满足：
- `WHERE b=?`（最左前缀缺失）
- `WHERE c=?`（同上）

### 11.4 关键索引一览

```sql
-- 高频
idx_orders_user_created   ON orders(user_id, created_at DESC);
idx_bt_user_created       ON balance_transactions(user_id, created_at DESC);

-- 部分（有 WHERE 子句）
idx_orders_expire_pending ON orders(expire_at) WHERE status='pending';
uniq_pay_order_success    ON payment_records(order_id) WHERE status='success';
idx_refund_pending        ON refund_records(order_id) WHERE status='pending';

-- 唯一
uniq_orders_no            ON orders(order_no);
uniq_orders_idem          ON orders(user_id, idempotency_key);
uniq_bt_event             ON balance_transactions(user_id, related_type, related_no, type);

-- GIN
idx_cigars_scenes_gin     ON cigars USING GIN (scenes);
```

## 12. 数据归档策略

### 12.1 哪些表会膨胀

| 表 | 月增量预估 | 90 天后 |
|---|---|---|
| `operation_logs` | ~10 万行 | ~30 万行 |
| `payment_callbacks` | ~5 万行 | ~15 万行 |
| `meituan_sync_logs` | ~10 万行 | ~30 万行 |
| `recommend_logs` | ~3 万行 | ~10 万行 |
| `admin_login_logs` | ~5 千行 | ~1.5 万行 |

`operation_logs.before_data/after_data` 是 JSONB，单行可达数 KB。半年到 1 年容易到几十 GB。

### 12.2 归档方案

```sql
-- 创建冷库表（同结构，单独 tablespace）
CREATE TABLE operation_logs_archive (LIKE operation_logs INCLUDING ALL);

-- 每月 1 号凌晨 03:00 跑：
INSERT INTO operation_logs_archive
  SELECT * FROM operation_logs
   WHERE created_at < now() - INTERVAL '90 days';

DELETE FROM operation_logs
 WHERE created_at < now() - INTERVAL '90 days';

VACUUM ANALYZE operation_logs;
```

可用 BullMQ 定时任务实现，详见 `06_部署与运维Runbook.md`。

### 12.3 应用层 4KB 截断

```ts
// src/common/utils/truncate.ts
export function truncateJson(obj: any, maxBytes = 4096): any {
  const json = JSON.stringify(obj);
  if (Buffer.byteLength(json) <= maxBytes) return obj;
  return { __truncated: true, sample: json.slice(0, maxBytes) + '...' };
}
```

## 13. 资金核对脚本

每日跑一次，验证不变量：

```sql
-- 每个用户的余额必须等于其流水净额
SELECT u.id, mp.balance_cents,
       COALESCE(SUM(bt.amount_cents * bt.direction), 0) AS sum_balance_txn
  FROM users u
  JOIN member_profiles mp ON mp.user_id = u.id
  LEFT JOIN balance_transactions bt ON bt.user_id = u.id
 GROUP BY u.id, mp.balance_cents
HAVING mp.balance_cents <> COALESCE(SUM(bt.amount_cents * bt.direction), 0);
```

返回任何行都是异常，立即电话告警。

## 15. 常见 DBA 检查清单

上线前 / 每月一次：

- [ ] 所有外键有索引
- [ ] 没有未使用的索引
- [ ] `pg_stat_activity` 无超长事务（> 5 分钟报警）
- [ ] `pg_stat_user_tables.n_dead_tup` 低（VACUUM 配置正常）
- [ ] WAL 归档正常
- [ ] 复制延迟 < 10 秒
- [ ] 连接数 < 80%
- [ ] 慢查询日志（> 500ms）每日检视
- [ ] 资金核对脚本无 diff
