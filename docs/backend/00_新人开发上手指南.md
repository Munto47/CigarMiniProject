# 00 · 新人开发上手指南

> **目标读者**：刚加入项目的后端工程师（包括第一次写后端的新手）
> **目标产出**：读完后能在 1 个工作日内跑通本地环境，3 个工作日内提交第一个 PR
> **配套阅读**：`Plan.md` → `01_项目结构与编码规范.md` → `03_关键场景实现指南.md`

## 0. 你需要先理解的 5 件事

1. **这是一个雪茄俱乐部的会员/电商系统**，分小程序前端 + React 管理后台 + Node.js 后端三层。我们做后端。
2. **核心是钱**：储值余额、积分、等级。任何动钱的操作必须**事务 + 流水表 + 唯一约束**三件套，少一件都会上线翻车。
3. **接 2 个外部系统**：微信支付（充值用）、美团收银（线下点单用）。它们的回调都可能丢失或重放，必须有兜底。
4. **用 NestJS + PostgreSQL + Redis + Prisma**。如果你只熟 Java，等价对照：NestJS ≈ Spring Boot；Prisma ≈ MyBatis-Plus；Decorator ≈ Annotation。
5. **不是从零开始**。前端代码与设计稿都在 `Cigar/`、`Cigar_admin/`、`Plan.md`、`docs/backend/` 里。**先读再写**。

### 1.3 初始化后端仓库（如果尚未创建）

```bash
# 在项目根目录
mkdir Cigar_server && cd Cigar_server
```

### 1.4 启动依赖中间件

在项目根目录创建 `docker-compose.dev.yml`：

```yaml
services:
  postgres:
    image: postgres:15-alpine
    environment:
      POSTGRES_USER: cigarpro
      POSTGRES_PASSWORD: cigarpro_dev
      POSTGRES_DB: cigarpro_dev
    ports:
      - "5432:5432"
    volumes:
      - pg_data:/var/lib/postgresql/data
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    command: ["redis-server", "--appendonly", "yes"]
    volumes:
      - redis_data:/data
  minio:
    image: minio/minio
    ports:
      - "9000:9000"
      - "9001:9001"
    environment:
      MINIO_ROOT_USER: minioadmin
      MINIO_ROOT_PASSWORD: minioadmin
    command: server /data --console-address ":9001"
    volumes:
      - minio_data:/data
volumes:
  pg_data: {}
  redis_data: {}
  minio_data: {}
```

启动：
```bash
docker compose -f docker-compose.dev.yml up -d
docker ps                          # 应看到 3 个容器 running
```

验证 PG 通畅：
```bash
docker exec -it cigarpro-postgres-1 psql -U cigarpro -d cigarpro_dev -c "SELECT 1;"
# 应输出: ?column? = 1
```

### 1.5 跑数据库 DDL

先启用扩展：
```bash
docker exec -it cigarpro-postgres-1 psql -U cigarpro -d cigarpro_dev <<EOF
CREATE EXTENSION IF NOT EXISTS btree_gist;
CREATE EXTENSION IF NOT EXISTS pgcrypto;
EOF
```

然后导入完整 DDL：
```bash
docker exec -i cigarpro-postgres-1 psql -U cigarpro -d cigarpro_dev \
  < ../docs/backend/02_数据库完整DDL.sql
```

验证：
```bash
docker exec -it cigarpro-postgres-1 psql -U cigarpro -d cigarpro_dev -c "\dt"
# 应看到 30+ 张表
docker exec -it cigarpro-postgres-1 psql -U cigarpro -d cigarpro_dev -c "SELECT COUNT(*) FROM permissions;"
# 应输出: 25
```

### 1.6 配置 Prisma

`Cigar_server/prisma/schema.prisma`（先用 `db pull` 从 DB 反向生成，再手动调整）：

```bash
# 先在 .env 配置 DATABASE_URL
echo 'DATABASE_URL="postgresql://cigarpro:cigarpro_dev@localhost:5432/cigarpro_dev"' > .env
echo 'REDIS_URL="redis://localhost:6379"' >> .env

# 反向生成 schema
pnpm prisma db pull
pnpm prisma generate
```

### 1.7 第一次启动

```bash
pnpm start:dev
```

打开浏览器：
- http://localhost:3000/health → 应返回 200
- http://localhost:3000/swagger → 应看到 Swagger UI

🎉 恭喜，环境通了。

## 3. 第三天：提交第一个 PR

### 3.1 选一个适合新手的任务

> 推荐入门任务（按难度递增）：
>
> 1. **修一个 SEED 数据**：把 `recommend_questions` 5 题录入（前端文档已列）。低风险，能熟悉 Prisma migrate。
> 2. **实现一个简单 GET 接口**：`GET /api/storedvalue/tiers`（返回所有启用的充值档位，公开接口）。
> 3. **实现一个 CRUD**：`/api/admin/library/tags` 风味标签 CRUD（带 RBAC + 操作日志）。
>
> **不要**第一周就去碰：支付、退款、等级计算、库存预占、回调处理。这些场景的复杂度需要先吃透 §3 再写。

### 3.2 开发流程

```bash
# 1. 同步主干
git switch main
git pull

# 2. 开新分支
git switch -c feat/recharge-tiers-list

# 3. 写代码（生成模块骨架）
pnpm dlx nest g module storedvalue
pnpm dlx nest g controller storedvalue
pnpm dlx nest g service storedvalue
```

### 3.3 写代码的 4 个铁律（来自 §0 全局原则）

1. **金额永远以「分」处理**。如果要返回元，加 `_yuan` 字段。
2. **写操作必须事务**。Service 里用 `prisma.$transaction(async (tx) => {...})`。
3. **写操作必须有幂等保护**。客户端发的 `Idempotency-Key`、DB 唯一约束。
4. **写操作必须写流水表**。改余额必写 `balance_transactions`，改积分必写 `point_transactions`，改等级必写 `level_change_logs`，禁止直接 UPDATE 业务表。

### 3.4 写测试

每个 PR 至少有：
- 1 个单元测试（Service 层）；
- 1 个接口测试（Controller 层，supertest）；
- 如果改了金额/积分/等级，再加 1 个并发测试（k6 或 Jest fake timer 模拟）。

```bash
pnpm test                          # 单元
pnpm test:e2e                      # 集成
pnpm test:cov                      # 覆盖率（要求 ≥ 80%）
```

### 3.5 提交 PR

```bash
git add .
git commit -m "feat(storedvalue): GET /api/storedvalue/tiers 返回启用档位"
git push -u origin feat/recharge-tiers-list
```

PR 描述模板：
```markdown
## 改动
- 实现 GET /api/storedvalue/tiers，返回启用且按 sort_order 排序的充值档位

## 影响范围
- 新增 storedvalue 模块
- 无 DDL 变更

## 测试
- [x] 单元测试覆盖 Service.findEnabledTiers
- [x] 接口测试 200 响应结构
- [x] Postman 联调通过

- 公开接口，但走限流（60 次/分钟/IP）
```

### 3.6 Code Review 后做什么

- **同意意见就改**。不要争"我觉得也可以"。
- **不同意要先理解**。如果 reviewer 说「这里要 FOR UPDATE」，先去 `docs/backend/03_关键场景实现指南.md §3` 读完，再回来回复。
- **改完再 push 一次**，commit message 用 `fix: address review comments`。

## 4. 第一周：把这些场景在脑子里跑一遍

读完代码不是会，**能在白板上画出流程**才是会。下面 6 个场景是后端最常被问的，准备能答到的程度：

### 4.1 用户充值 100 元，钱怎么进来的？

```
小程序点击充值按钮 100 元档位
    │
    ├─→ POST /api/member/recharge { tierId: 2 }
    │   后端: 创建 RechargeOrder(pending, expire_at=now+30min)
    │         调微信支付 v3 下单 API → 拿到 prepay_id
    │         返回 payParams 给小程序
    │
小程序 wx.requestPayment(payParams)
    │
微信支付完成 → 微信平台 POST /api/payment/wechat-callback
    │
    ├─→ 验签（RSA-SHA256，失败 401 不入业务）
    ├─→ AES-256-GCM 解密 ciphertext
    ├─→ 写 payment_callbacks（唯一约束兜底重放）
    ├─→ 业务事务:
    │     SELECT recharge_orders FOR UPDATE NOWAIT
    │     幂等命中 status='success' → 直接 200
    │     UPDATE member_profiles balance_cents += 10000, recharge_points += 100
    │     INSERT balance_transactions (recharge, +10000)  -- 唯一约束兜底
    │     INSERT point_transactions   (recharge_earn, +100)
    │     可能 INSERT level_change_logs
    │     UPDATE recharge_orders status='success'
    └─→ 200 + {"code":"SUCCESS"}
```

回调如果丢了？→ cron 兜底（§7.10）。

### 4.2 用户下单 200 元用余额支付，库存怎么不卖超？

详见 `Plan.md §7.7`。要点：
- 创建订单时 `stock_locked += qty`，可售 = `stock - stock_locked`，CHECK 约束兜底；
- 支付成功时 `stock -= qty, stock_locked -= qty`；
- 取消/超时时 `stock_locked -= qty`；
- 100 库存遇到 200 个并发请求，PG 行级锁会让其中 100 个看到 `stock - stock_locked < qty` 而失败（3002）。

### 4.3 同一个用户在 2 秒内连续点 2 次「立即支付」，怎么不重复扣款？

详见 `Plan.md §7.2 + §7.5`。要点：
- 第一次点击：`SELECT FOR UPDATE NOWAIT` 锁住订单，扣余额，标 `paid`；
- 第二次点击（2 秒后）：`SELECT FOR UPDATE NOWAIT` 命中（第一次事务已提交），但 `status='paid'`，走幂等命中分支，返回上次结果；
- 如果第二次正好与第一次同时进入：`NOWAIT` 让其中一个直接失败（2004），前端重试时已是 `paid`，进幂等分支。

### 4.4 管理员对一个用户「手动加 100 元」，怎么防双击？

详见 `Plan.md §7.5` 第 5 行。要点：
- 必须带 `Idempotency-Key`（前端生成 UUID）；
- 后端 Redis SETNX `idem:adjust:{adminId}:{key}` TTL 10min；
- DB 层兜底：`balance_transactions` 唯一约束 `(userId, 'manual', key, 'adjust')`；
- 双击时第二次 Redis 命中，直接返回上次 transactionId。

### 4.5 一笔 200 元订单退 80 元，状态怎么变？

- `orders.refunded_amount_cents = 80`（事务内 += 80）；
- `orders.status` 保持 `completed`（部分退）；
- `refund_records` 新增一条 `success`；
- `balance_transactions` 一条 `refund +80`；
- `point_transactions` 一条 `consume_revoke -80`；
- 如果累计退到 200，`orders.status='refunded'`。
- 退款进行中再次发起 → 4005。

### 4.6 美团回调 5 分钟没来怎么办？

详见 `Plan.md §7.10`。
- cron 每 5 分钟扫 `orders WHERE status='settling' AND paid_at < now() - 10min`，调美团查询接口；
- 确认已支付完成 → 推进到 `completed`；
- 如果美团那边也没找到，写 `out_of_sync` 状态，告警人工。

## 5. 常见错误与避坑

| 现象 | 原因 | 解决 |
|---|---|---|
| `psql: FATAL: role "cigarpro" does not exist` | docker compose 没起来或环境变量没生效 | `docker compose down -v && docker compose up -d` |
| Prisma migrate 报 `EXCLUDE USING GIST` 不识别 | 没启用 btree_gist 扩展 | 先 `CREATE EXTENSION btree_gist;` |
| `balance_cents` 变成负数 | 没用 `WHERE balance_cents >= $amount` 校验 + CHECK 约束没启用 | DDL 必须有 `CHECK (balance_cents >= 0)`，UPDATE 必须有 WHERE 条件 |
| 同一笔充值入账 2 次 | 没有写 `balance_transactions` 唯一约束 | 见 `Plan.md §3.2 修订 1` |
| 测试时 fake timer 不前进 | Jest 的 `jest.useFakeTimers()` 与 BullMQ 冲突 | 单测把 BullMQ mock 掉，集成测才用真队列 |
| 接口报 423/2004 | NOWAIT 没拿到锁 | 客户端重试，或缩短上一个事务时间 |
| 退款金额累加错乱 | 没用 `refunded_amount_cents` 字段，靠 SUM 算 | 必须用字段，事务内 += |
| Redis 过期后幂等失效 | TTL 太短或 Redis 重启 | TTL ≥ 10 min；DB 唯一约束兜底 |
